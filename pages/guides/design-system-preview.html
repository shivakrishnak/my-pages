PKH Design System

A complete living reference for shell.css v3 — every token, component, and usage pattern for the Personal Knowledge Hub. One file powers every page.

01What is shell.css?

A single shared CSS file that defines the design language for every page in the PKH. It contains all tokens, layout primitives, navigation chrome, and reusable components. Pages add a <link> to it and focus only on their own content.

Core principle: The UI disappears behind the content. Ground is paper. Text is ink. Chrome is invisible.
02Quick setup — any page
HTML boilerplate 
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Page Title</title>
</head>

  </style>
</body>
</html>

04Dark mode

Dark mode is activated by adding the .dark class to <html> or <body>. init.js handles this before first paint to prevent flash. All tokens cascade automatically — no per-component overrides needed.

js
// Toggle dark mode
document.documentElement.classList.toggle('dark');

// Set on load (init.js does this automatically)
if (localStorage.getItem('theme') === 'dark')
  document.documentElement.classList.add('dark');
05Event delegation rule (critical)
⚠ Always use event delegation. Because init.js rebuilds body.innerHTML after page scripts first parse, upfront NodeList queries point to detached nodes. Always use document.addEventListener('click', e => e.target.closest('.selector')) — never querySelectorAll at parse time.

05Accordion
What is the most effective learning technique?
Active recall — forcing yourself to retrieve information from memory — is consistently the most effective technique across decades of cognitive science research. It outperforms re-reading, highlighting, and summarizing by a factor of 3×.
How does spaced repetition work?
Spaced repetition exploits the spacing effect — information reviewed at increasing intervals is retained far longer than information reviewed multiple times in a single session.

Rules & Patterns

Non-negotiable rules that keep the system consistent as it grows to hundreds of pages.

01The golden rules
  1. Never fork shell.css. Page-specific styles go in a <style> block on the page. No exceptions.
  2. Always use event delegation. document.addEventListener('click', e => e.target.closest('.sel')) — never querySelectorAll at parse time.
  3. Use tokens, not hardcoded values. Write var(--indigo), never #4f46e5 in page styles.
  4. Mobile first. Every new component must work at 320px. Test at 320/375/430/768/desktop.
  5. No inline styles for layout. Use the grid and spacing token classes. Inline styles are only for demo/preview content.
  6. Run the registry generator after every new page or rename. The homepage is data-driven, not hardcoded.
  7. Add new tokens to section 2 of shell.css and the header reference comment simultaneously.
02Naming conventions
Type Pattern Example
Shell classes .ms- prefix .ms-card, .ms-btn
CSS variables -- prefix, hyphen-separated --indigo-lt, --space-4
Page IDs Descriptive, lowercase hyphen id="s1-active-recall"
Data attributes data- prefix on body data-layout="full"
Legacy (tracker) No prefix .card, .badge, .check-row
03When to extend vs. add to shell.css
Extend in page <style>
One-off layout tweaks. Page-specific hero patterns. Custom grid variations. Component sizing specific to one page.
Add to shell.css
Components used on 2+ pages. New semantic color. New layout utility. New interactive pattern that will recur.
04Accessibility checklist
  • Every interactive element has visible focus state (:focus-visible is in shell.css)
  • All images have descriptive alt text
  • Heading hierarchy is correct: one h1, h2s for sections, h3 for sub-sections
  • Touch targets are at least 44×44px (check-rows, nav items, buttons)
  • Color is never the only differentiator (badges always have text labels)
  • Skip link present (init.js injects it automatically)
  • Page tested with keyboard navigation only
