Hub Documentation

How to Add a New Page or Project

Everything needed to add a single page, start a whole new project group, and fix the most common publishing mistake — a page with no styling applied.

§1How pages share their UI

Every page in this hub links exactly one file: core/init.js. That single script injects everything else — shell.css, the sidebar, topbar, mobile bottom-nav, dark mode applied before first paint, reading progress, and scroll-spy — around whatever content you put inside a <div id="page-content"> wrapper.

You never hand-write shell HTML (sidebar markup, topbar markup, bottom-nav markup) on a page. If you find yourself copying that structure from another page, stop — that's a sign you're working against the architecture, not with it.

The rule of thumb: never fork or copy shell.css, and never hand-write the shell's sidebar/topbar/bottom-nav HTML. If a page needs layout shell.css doesn't provide, add a page-local <style> block on top of it — exactly like the homepage does for its card grid.

What's not shared is content layout. The homepage looks like a card grid because it has its own <style> block. A reference doc page looks like a reading page because it uses shell.css's section and callout classes. Both pull from the same palette and the same nav chrome — the shell — but content layout is chosen per page.

§2The layout system — data-layout

Every page declares its desired shell experience with one attribute on <body>:

Value Gives you Use for
full (default) Sidebar + topbar + bottom-nav Documentation, mastery pages, reference systems, long-form content
mixed Topbar + bottom-nav, no sidebar Tools, dashboards, trackers, calculators, single-screen utilities
none Nothing injected — fully self-contained Highly custom or experimental pages. Use sparingly.

Two templates in core/templates/ correspond to the two common cases:

Template data-layout Use it when…
template-standalone.html mixed Single-screen tool, calculator, dashboard, tracker — no jump-to-section navigation needed
template-with-sidebar.html full Long-form reference material with multiple sections — like this guide, or the mastery system pages
Not sure which one? Ask: "Would this page benefit from a clickable table of contents?" If yes, use full. If the whole page fits on one or two screens, use mixed — a sidebar with almost nothing in it looks broken, not thorough.

Getting sidebar content — two options

If you use data-layout="full", the sidebar needs content. Two ways to provide it:

  • Automatic: add data-toc="true" to <body>. init.js scans this page's own h2/h3 headings that have id attributes and builds a table of contents automatically. Best for a single standalone page like this one.
  • Custom: add a <div id="page-sidebar"> anywhere in <body> containing hand-written .ms-nav-item links. init.js uses this verbatim instead of auto-building a TOC. Use this whenever the sidebar needs more than this page's own headings — a multi-page project's "All Pages In This Project" list (see core/templates/template-with-sidebar.html), or tab-based navigation (see the workout plan, which uses a custom sidebar to mirror its day-tabs).

§3How grouping works

Pages are grouped into projects by folder, not by filename prefix. Every file inside pages/mastery/ is automatically grouped under "Mastery System" on the homepage, with zero configuration — the registry script derives the category from the folder name. 

pages/
├── mastery/              → groups as "Mastery System" (overridden per-file, see below)
│   ├── part1.html
│   └── ...
├── hub/                   → groups as "Hub" automatically
│   └── credit-card-hub.html
├── guides/                → groups as "Guides" automatically
│   └── how-to-add-a-page.html   (this page)
└── your-next-project/     → would group as "Your Next Project" automatically
    └── whatever.html

Folder name becomes category name automatically, title-cased. Override it any time with an explicit <meta name="page-category"> tag in a specific file — that's how 90day-plan.html shows under "Trackers & Tools" even though it physically lives in the mastery/ folder alongside the reference docs.

§4Step by step — add a single page to an existing project

Example: adding one more page to the mastery/ group.

  1. Copy whichever template fits — cp core/templates/template-with-sidebar.html pages/mastery/your-new-page.html
  2. Open it and replace every block marked ░░░ EDIT ░░░ — title, description, data-layout/data-title/data-brand attributes, sidebar links, content sections.
  3. If using a custom #page-sidebar "All Pages In This Project" list, add a line for this new page, and go back and add a matching line to the other pages in mastery/ so they all link to the new one too.
  4. Confirm the script path is correct for the folder depth — a file at pages/mastery/your-new-page.html needs <script src="../../core/init.js"> (two levels up to reach the repo root, then into core/). See §7 below if this is wrong.
  5. If the page has any interactivity, write its <script> block outside #page-content but still inside <body>, and use event delegation — see §9.
  6. Run the registry generator from the repo root: 
    python core/generate_registry.py
  7. Check the script's printed output — confirm your new page is listed under "Mastery System" with the right title.
  8. Commit and push. Done.

§5Step by step — start a brand new project group

Example: starting a "Finance Tracker" project with a totally different look from the mastery docs.

  1. Create the folder: mkdir pages/finance-tracker
  2. Decide: does this need a sidebar, or is it a single dashboard screen? For a dashboard, copy template-standalone.html (already set to data-layout="mixed"). 
    cp core/templates/template-standalone.html pages/finance-tracker/index.html
  3. Edit the marked sections — title, description, data-title, content.
  4. If this page needs a layout shell.css doesn't provide (a chart grid, a budget table, whatever), add it in the page's own <style> block, the way the homepage does for its card grid. Use shell.css color variables (var(--indigo), var(--surface), var(--border), etc.) so it still matches the rest of the hub in both light and dark mode.
  5. Run python core/generate_registry.py from the repo root.
  6. Check the output — you should see a new category, "Finance Tracker", appear automatically. No homepage edits needed.
  7. Commit and push.
This is the entire value of the folder-based grouping system: a brand new, visually distinct project type takes zero changes to index.html, shell.css, or the registry script. You only ever touch files inside your new project's own folder.

§6Metadata reference

These go in the <head> (registry metadata) or on <body> (layout metadata) of any page. Only <title> is required — everything else has a sensible default.

Registry metadata (in <head>)

Tag Required? Default if omitted
<title> Yes Filename, title-cased
<meta name="description"> No Empty — card shows "Click to open"
<meta name="page-icon"> No 📄
<meta name="page-category"> No Parent folder name, title-cased ("General" if no subfolder)
<meta name="page-order"> No Alphabetical by title within category
<meta name="page-hidden"> No Page is visible; set to "true" to hide from homepage without deleting it

Layout metadata (on <body>)

Attribute Required? Default if omitted
data-layout No full
data-title No Falls back to <title>
data-brand No "Shiva's Knowledge Hub"
data-badge No No badge shown
data-badge-color No indigo — also accepts teal, amber, coral, sage
data-toc No "false" — sidebar shows a placeholder unless a custom #page-sidebar is present

§7Fixing: page shows as plain text, no CSS

This is almost always one of three causes, in order of likelihood:

1. Wrong relative path to init.js

The number of ../ in the script tag must match how deep the file sits below the repo root. Check this first — it's the most common cause.

File location Correct path to init.js
pages/page.html ../core/init.js
pages/mastery/part1.html ../../core/init.js
pages/mastery/sub/deep.html ../../../core/init.js

To check fast: open the page live, open browser dev tools (F12), go to the Network tab, reload, and look for init.js and shell.css — if either shows a red 404, the path is wrong.

2. Case mismatch

GitHub Pages serves files from Linux, which is case-sensitive. Core/Init.js is a different file from core/init.js as far as the server is concerned, even though most local setups (Mac, Windows) won't show this as an error while you're testing locally. Always use lowercase exactly matching the real folder and file names.

3. The <script> tag is missing entirely

If the page was created by pasting raw content rather than starting from one of the two templates in core/templates/, it may simply never have had init.js linked. Compare the page's <head> against a working page like pages/hub/credit-card-hub.html — the first 10–12 lines should look essentially identical except for the title and meta tags.

Fastest diagnosis: view the page's source (right-click → View Page Source, or Ctrl/Cmd+U) and check the <head> for <script src="...init.js">. If it's missing, that's cause #3. If it's present, copy the exact src value and check it against the table above for cause #1.

§8Fixing: page doesn't appear on the homepage

  1. Confirm the file actually has a .html extension and sits somewhere under pages/ (any depth — the scan is recursive).
  2. Confirm python core/generate_registry.py was actually run after adding the file — the homepage reads a generated JSON snapshot, not the live folder, so a new file with no registry rebuild simply won't show up yet.
  3. Check the script's printed output — it lists every page it found. If yours isn't listed, check for a stray <meta name="page-hidden" content="true"> tag that might be hiding it intentionally.
  4. Open core/registry.json directly and confirm your page's entry is actually there with the path you expect.

§9Fixing: my page's buttons/tabs stopped working

This is the one bug that's unique to this architecture and worth understanding, not just fixing. init.js completely rebuilds <body>'s DOM after your page's own scripts have already run once (to collect and move them). Any querySelectorAll result or element reference captured before that rebuild is now a detached node — it still exists in memory, but it's disconnected from the visible page, so click handlers on it silently do nothing.

Broken pattern 
document.querySelectorAll('.tab').forEach(function (t) {
  t.addEventListener('click', handleClick);
});
This attaches listeners to the tabs that existed at parse time. After init.js rebuilds the DOM, the visible tabs are new elements — the listeners above are attached to nothing anyone can click.
Correct pattern — event delegation 
document.addEventListener('click', function (e) {
  var tab = e.target.closest('.tab');
  if (!tab) return;
  handleClick(tab);
});
This attaches one listener to document itself, which never gets replaced. It checks on every click whether the thing clicked was (or was inside) a .tab. Works correctly no matter how many times the DOM around it gets rebuilt.

See pages/hub/credit-card-hub.html for a real, committed example of interactive JS built entirely with delegation.

§10Pre-publish checklist

  • Started from one of the two templates in core/templates/, not a blank file
  • <title> and <meta name="description"> are filled in and accurate
  • init.js script path has the correct number of ../ for this file's folder depth
  • Content is wrapped in <div id="page-content">, not left loose in <body>
  • Any page-specific <script> uses event delegation, not querySelectorAll + forEach
  • Ran python core/generate_registry.py from the repo root after adding the file
  • Checked the page in a browser — sidebar/topbar render with the correct shell.css colors, not plain unstyled HTML
  • If part of a multi-page project using a custom #page-sidebar, every sibling page's "All Pages" list was updated to include the new page
