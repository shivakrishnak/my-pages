How to Add a New Page or Project

Everything needed to add a single page, start a whole new project group, and fix the most common publishing mistake — a page with no styling applied.

§1How pages share their UI

Every page in this hub — the homepage and every content page — links the same two files: core/shell.css and core/shell.js. That gives every page, for free: the sidebar, topbar, mobile bottom-nav, dark mode, color palette, typography, and about 30 reusable content components (callout boxes, tables, framework cards, step lists, and more).

What's not shared is layout. The homepage looks like a card grid because it has its own <style> block on top of shell.css for that grid. A reference doc page looks like a reading page with a sidebar because it uses shell.css's sidebar component classes. Both pull from the same palette and the same nav chrome, but the layout is chosen per page or per project group.

The rule of thumb: never fork or copy shell.css into a project folder. If a page needs layout shell.css doesn't provide, add a page-local <style> block on top of it — exactly like the homepage does for its card grid.

§2The two page templates

Two ready-to-copy starting points live in core/templates/. Pick based on how much content the page has, not based on what "feels more proper."

Template Use it when… What you get
template-standalone.html Single-screen tool, calculator, dashboard, short note — anything that doesn't need jump-to-section navigation Topbar only, with a home link built into it. No sidebar, no dead margin.
template-with-sidebar.html Long-form reference material with multiple sections — like this guide, or the mastery system pages Full sidebar with table of contents, search-within-page, and a multi-page project list
Not sure which one? Ask: "Would this page benefit from a clickable table of contents?" If yes, use the sidebar template. If the whole page fits on one or two screens, use standalone — a sidebar with three items in it looks broken, not thorough.

§3How grouping works

Pages are grouped into projects by folder, not by filename prefix. Every file inside pages/mastery/ is automatically grouped under "Mastery System" on the homepage, with zero configuration — the registry script derives the category from the folder name. 

pages/
├── mastery/              → groups as "Mastery System" (overridden per-file, see below)
│   ├── part1.html
│   └── ...
├── guides/                → groups as "Guides" automatically
│   └── how-to-add-a-page.html   (this page)
└── your-next-project/     → would group as "Your Next Project" automatically
    └── whatever.html

Folder name becomes category name automatically, title-cased. pages/finance-tracker/ would become "Finance Tracker" on the homepage with no extra steps. Override it any time with an explicit <meta name="page-category"> tag in a specific file — that's how 90day-plan.html shows under "Trackers & Tools" even though it physically lives in the mastery/ folder alongside the reference docs.

§4Step by step — add a single page to an existing project

Example: adding one more page to the mastery/ group.

  1. Copy whichever template fits — cp core/templates/template-with-sidebar.html pages/mastery/your-new-page.html
  2. Open it and replace every block marked ░░░ EDIT ░░░ — title, description, sidebar TOC links, content sections.
  3. In the sidebar's "All Pages In This Project" list, add a line for this new page, and go back and add a matching line to the other pages in mastery/ so they all link to the new one too.
  4. Confirm the asset paths are correct for the folder depth — a file at pages/mastery/your-new-page.html needs ../../core/shell.css (two levels up to reach the repo root, then into core/). See §7 below if this is wrong.
  5. Run the registry generator from the repo root: 
    python core/generate_registry.py
  6. Open index.html locally or check the console output of the script — confirm your new page is listed under "Mastery System" with the right title.
  7. Commit and push. Done.

§5Step by step — start a brand new project group

Example: starting a "Finance Tracker" project with a totally different look from the mastery docs.

  1. Create the folder: mkdir pages/finance-tracker
  2. Decide: does this need a sidebar, or is it a single dashboard screen? For a dashboard, copy template-standalone.html. 
    cp core/templates/template-standalone.html pages/finance-tracker/index.html
  3. Edit the marked sections — title, description, home link, content.
  4. If this page needs a layout shell.css doesn't provide (a chart grid, a budget table, whatever), add it in the page's own <style> block, the way the homepage does for its card grid. Use shell.css color variables (var(--indigo), var(--surface), var(--border), etc.) so it still matches the rest of the hub in both light and dark mode.
  5. Run python core/generate_registry.py from the repo root.
  6. Check the output — you should see a new category, "Finance Tracker", appear automatically. No homepage edits needed.
  7. Commit and push.
This is the entire value of the folder-based grouping system: a brand new, visually distinct project type takes zero changes to index.html, shell.css, or the registry script. You only ever touch files inside your new project's own folder.

§6Registry metadata reference

These go in the <head> of any page. Only <title> is required — everything else has a sensible default.

Tag Required? Default if omitted
<title> Yes Filename, title-cased
<meta name="description"> No Empty — card shows "Click to open"
<meta name="page-icon"> No 📄
<meta name="page-category"> No Parent folder name, title-cased ("General" if no subfolder)
<meta name="page-order"> No Alphabetical by title within category
<meta name="page-hidden"> No Page is visible; set to "true" to hide from homepage without deleting it

§7Fixing: page shows as plain text, no CSS

This is almost always one of four causes, in order of likelihood:

1. Wrong relative path to shell.css

The number of ../ in the stylesheet link must match how deep the file sits below the repo root. Check this first — it's the most common cause.

File location Correct path to shell.css
pages/page.html ../core/shell.css
pages/mastery/part1.html ../../core/shell.css
pages/mastery/sub/deep.html ../../../core/shell.css

To check fast: open the page live, open browser dev tools (F12), go to the Network tab, reload, and look for shell.css — if it shows a red 404, the path is wrong. Fix the href in the <link> tag to match the table above.

2. Case mismatch

GitHub Pages serves files from Linux, which is case-sensitive. Core/Shell.css is a different file from core/shell.css as far as the server is concerned, even though most local setups (Mac, Windows) won't show this as an error while you're testing locally. Always use lowercase exactly matching the real folder and file names.

3. The <link> tag is missing entirely

If the page was created by pasting raw content rather than starting from one of the two templates in core/templates/, it may simply never have had the stylesheet link added. Compare the page's <head> against a working page like any file in pages/mastery/ — the first 15–20 lines should look essentially identical except for the title and meta tags.

4. .nojekyll missing from the repo root

GitHub Pages runs files through Jekyll by default unless a file named exactly .nojekyll (with the leading dot) sits in the repo root. Without it, Jekyll can skip or mangle certain files. Confirm it exists: ls -la in the repo root should show it. If it's missing, create an empty file with that exact name and commit it.

Fastest diagnosis: view the page's source (right-click → View Page Source, or Ctrl/Cmd+U) and check the very first few lines for the <link rel="stylesheet" href="..."> tag. If it's missing, that's cause #3. If it's present, copy the exact href value and check it against the table in this section for cause #1.

§8Fixing: page doesn't appear on the homepage

  1. Confirm the file actually has a .html extension and sits somewhere under pages/ (any depth — the scan is recursive).
  2. Confirm python core/generate_registry.py was actually run after adding the file — the homepage reads a generated JSON snapshot, not the live folder, so a new file with no registry rebuild simply won't show up yet.
  3. Check the script's printed output — it lists every page it found. If yours isn't listed, check for a stray <meta name="page-hidden" content="true"> tag that might be hiding it intentionally.
  4. Open core/registry.json directly and confirm your page's entry is actually there with the path you expect.

§9Pre-publish checklist

  • Started from one of the two templates in core/templates/, not a blank file
  • <title> and <meta name="description"> are filled in and accurate
  • Stylesheet and script paths have the correct number of ../ for this file's folder depth
  • Page has a working link back to the homepage (sidebar footer, or the topbar home link on the standalone template)
  • Ran python core/generate_registry.py from the repo root after adding the file
  • Checked the page in a browser — sidebar/topbar render with the correct shell.css colors, not plain unstyled HTML
  • If part of a multi-page project, every sibling page's "All Pages" list was updated to include the new page
